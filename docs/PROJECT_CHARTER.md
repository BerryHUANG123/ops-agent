# OpsAgent 项目立项文档

> 版本：V1.0 | 更新日期：2026-05-23 | 状态：已批准

---

## 一、项目背景

Linux 服务器运维长期依赖人工巡检模式，存在以下痛点：

| 痛点 | 表现 | 影响 |
|------|------|------|
| **效率低** | 运维人员需逐台登录、手动执行命令检查 | 单次巡检耗时 30min+，多机环境线性增长 |
| **遗漏多** | 依赖经验判断，缺乏系统化检测覆盖 | 磁盘满、内存泄漏等问题常在故障后才被发现 |
| **响应慢** | 发现问题后需人工分析、手动修复 | 平均故障恢复时间（MTTR）数小时 |
| **成本高** | 7×24 值班人力投入大 | 中小团队难以承受专职运维成本 |

当前市场上 Zabbix、Prometheus 等方案虽成熟，但部署复杂、资源占用高、学习曲线陡峭，对单机或小规模场景显得"杀鸡用牛刀"。

---

## 二、应用价值

OpsAgent 定位为 **轻量级、智能化、低功耗** 的 Linux 服务器自治运维助手：

- **7×24 无人值守** — 后台常驻守护，无需人工盯守
- **故障秒级发现** — 每 30 秒采集一次系统指标，异常即时触发处置
- **自动修复闭环** — 常见问题（磁盘满、僵尸进程、服务崩溃）自动执行预定义修复动作
- **减少人工 80%** — 从"人盯机器"转变为"机器自治、人管策略"
- **零外部依赖** — 不需要数据库服务器、消息队列，单文件即可运行
- **低资源占用** — 内存 < 50MB，CPU < 1%，适合低功耗设备（J1900/N100/树莓派）

## 三、适用场景

### 3.1 主要场景

| 场景 | 说明 | 典型用户 |
|------|------|----------|
| 单机 Linux 服务器 | 一台机器跑 Web/数据库/文件服务 | 个人站长、小型企业 |
| 多机 Linux 集群 | 10 台以内服务器统一监控 | 中小团队、SaaS 创业公司 |
| 容器化服务 | Docker 容器状态监控 | 微服务架构运维 |
| Web 服务监控 | HTTP/HTTPS 站点可用性检测 | 网站运维、API 网关 |
| 边缘设备 | 低功耗 ARM/x86 设备 | IoT 网关、NAS、家庭服务器 |

### 3.2 不适用场景

- 超大规模集群（100+ 节点）— 建议使用 Prometheus + Grafana
- 需要复杂拓扑发现的网络监控 — 建议使用 Zabbix
- Windows 服务器监控 — 超出 V1.0 范围

## 四、项目目标

### 4.1 V1.0 — 基础监控 + 简单处置（当前版本）

**核心目标：跑起来、查得到、修得了**

- [x] 系统指标采集：CPU、内存、磁盘、网络、进程
- [x] 阈值检测与告警：可配置规则、多级告警
- [x] 自动执行器：预定义修复脚本（清理磁盘、重启服务等）
- [x] 轻量记忆：SQLite 存储事件历史，支持查询
- [x] HTML 报表：每日汇总报告，可选发送
- [x] 调度引擎：统一管理采集、检测、报表周期
- [x] systemd 集成：开机自启、服务管理

**验收标准：**
- 单机部署 < 5 分钟
- 内存占用 < 50MB
- CPU 占用 < 1%（平均）
- 故障检测延迟 < 60 秒
- 自动修复成功率 > 90%（预定义场景）

### 4.2 V2.0 — 智能分析 + 记忆优化（规划中）

**核心目标：能分析、会学习、更聪明**

- [ ] AI 辅助分析：接入 LLM 进行日志分析、根因推断
- [ ] 自适应阈值：基于历史数据动态调整告警阈值
- [ ] 记忆优化：向量化存储、语义检索历史事件
- [ ] 多机协同：Agent 间通信、集中式 Dashboard
- [ ] Web UI：浏览器端管理界面
- [ ] 插件系统：支持用户自定义检测器和执行器

## 五、技术架构概要

```
┌─────────────────────────────────────────────┐
│                  调度引擎 (Scheduler)         │
│          统一管理所有模块的执行周期            │
├─────────┬─────────┬─────────┬───────────────┤
│  采集器  │  检测器  │ 执行器  │   记忆模块     │
│Collector │Detector │Executor │   Memory      │
│ psutil   │ 规则引擎 │ 脚本库  │   SQLite      │
├─────────┴─────────┴─────────┴───────────────┤
│                 报表模块 (Reporter)           │
│            Jinja2 模板 → HTML 报告            │
└─────────────────────────────────────────────┘
```

**技术栈：**
- 语言：Python 3.10+
- 采集：psutil
- 存储：SQLite（内嵌，零配置）
- 模板：Jinja2
- 部署：systemd service

## 六、风险评估与应对

| 风险 | 等级 | 概率 | 影响 | 应对措施 |
|------|------|------|------|----------|
| **自动修复误操作** | 🔴 高 | 中 | 高 | 1. 默认仅执行只读检测，写操作需显式启用<br>2. 所有修复动作记录审计日志<br>3. 支持 dry-run 模式预览 |
| **SQLite 并发瓶颈** | 🟡 中 | 低 | 中 | 1. 单线程写入设计<br>2. WAL 模式提升读并发<br>3. V2.0 可迁移至 PostgreSQL |
| **psutil 跨发行版兼容** | 🟡 中 | 低 | 低 | 1. 目标 CentOS/Ubuntu/Debian 三大主流发行版<br>2. CI 矩阵测试<br>3. 降级处理不支持的指标 |
| **资源占用超标** | 🟡 中 | 低 | 中 | 1. 采集间隔可配置（默认 30s）<br>2. 内存使用限制告警<br>3. 历史数据自动清理（默认 30 天） |
| **规则配置错误导致漏报** | 🟡 中 | 中 | 中 | 1. 提供开箱即用的默认规则集<br>2. 配置校验工具<br>3. 定期规则审计提醒 |
| **单点故障** | 🟢 低 | 低 | 低 | 1. systemd 自动重启<br>2. 崩溃恢复后自动补齐检测<br>3. V2.0 支持多机冗余 |
